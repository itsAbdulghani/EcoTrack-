1. Introduction

1.1 Purpose and Scope

The Environmental Monitoring System (EMS) is a comprehensive, enterprise-grade solution designed to provide real-time monitoring, data logging, and alarming for critical environmental parameters in pharmaceutical, biotechnology, and medical device manufacturing environments. This system ensures compliance with Good Manufacturing Practices (GMP) and regulatory requirements by providing validated, secure, and reliable environmental monitoring capabilities.

Primary Functions

Real-time Monitoring

Temperature, humidity, particulate matter, and other critical parameters

Continuous Data Logging

With timestamp validation and audit trail

Configurable Alarm System

Thresholds, notifications, and escalation procedures

Data Integrity

Electronic signatures, audit trails, and secure access

Target Audience

This manual provides detailed instructions for system administrators, operators, and quality assurance personnel responsible for the operation, maintenance, and validation of the Environmental Monitoring System.

2.2 Data Flow and Communication

The system's data flow follows a structured process ensuring data integrity and reliability from sensor acquisition through to storage and alarm processing.

1. Data Acquisition

The Windows Service polls sensors at configurable intervals (default: 60 seconds) via industrial communication protocols.

2. Data Processing

Raw sensor values undergo scaling and offset calculations to produce calibrated, engineering-unit values.

3. Data Storage

Processed values are stored in the database with timestamp, sensor ID, status, and calibration data.

4. Alarm Processing

Values outside threshold ranges trigger alarms with delay timers, physical activation, and notifications.

Data Processing Formulas

Calibrated Value Calculation

// Calibrated Value = Raw Value + Offset CalibratedValue = RawValue + Offset

Final Value Scaling

// Scale raw input to engineering units FinalValue = ((CalibratedValue - InputMin) / (InputMax - InputMin)) * (ScaleMax - ScaleMin) + ScaleMin

7. Alarm Handling & Notifications

Alarm configuration, prioritization, and response procedures

7.1 Alarm Configuration

Low Alarm Configuration
  • Set Low Alarm Limit below normal operating range
  • Alarm triggers when sensor value drops below this limit
  • Recommended hysteresis of at least 0.5 units above the limit
  • Example: Temperature sensor with Low Alarm Limit of 18C for room with normal range 20-25C
High Alarm Configuration
  • Set High Alarm Limit above normal operating range
  • Alarm triggers when sensor value exceeds this limit
  • Recommended hysteresis of at least 0.5 units below the limit
  • Example: Humidity sensor with High Alarm Limit of 65%RH for room with normal range 45-60%RH
Creating Alarm Thresholds
  1. Navigate to Sensors > Edit Sensor
  2. Locate Alarm Configuration section
  3. Enter Low Alarm Limit (or leave blank for no low alarm)
  4. Enter High Alarm Limit (or leave blank for no high alarm)
  5. Set Alarm Delay in seconds (0-3600)
  6. Save the configuration

Alarm Delay: Prevents spurious alarms from temporary fluctuations. Example: Set 30-second delay for temperature sensors to avoid alarms from brief HVAC cycles.

7.2 Alarm Prioritization

7.4 Alarm Acknowledgment and Resolution

Acknowledging Alarms
  1. Navigate to Alarms > Alarm Log
  2. Locate the active alarm requiring acknowledgment
  3. Click the "Acknowledge" button next to the alarm
  4. Enter acknowledgment message (optional but recommended)
  5. Click "Confirm" to acknowledge the alarm

Requirements: Only users with "Alarm Acknowledge" permission can acknowledge alarms. Acknowledgment is logged in audit trail with username and timestamp.

Alarm Resolution Process
  1. Identify the root cause of the alarm condition
  2. Take corrective and preventative actions
  3. Once sensor reading is back within limits, alarm will be marked as "Resolved"
  4. Manually close the alarm by clicking "Resolve Alarm"
  5. Enter resolution details and any corrective actions taken
  6. Require electronic signature for final resolution approval
Resolution Documentation Requirements

All resolution details are permanently stored in the alarm log. The following documentation is required:

  • Root cause analysis and investigation findings
  • Corrective and preventive actions taken
  • Link to any related change control or deviation forms
  • Electronic signature for approval and closure
  • Mandatory audit trail entry with full documentation

8. Data Logging & Reporting

Data collection, storage, reporting, and export capabilities

8.1 Data Collection Process

Logging Interval

Data points collected every 60 seconds (default). Logging interval is globally configurable in System Settings.

Time Stamping

All logged data includes unique, unalterable UTC timestamps with millisecond precision.

Data Validation

Data validated against pre-set ranges before storage. Out-of-range data logged with error flag.

8.2 Data Storage and Integrity

Database Structure
  • Relational database (Microsoft SQL Server) for data security
  • Separated tables for: Sensor Data, Alarm Log, Audit Trail, Configuration
  • Indexes optimized for rapid querying and reporting
  • Transaction logging enabled for data recovery
Data Retention Policy
  • Historical data retained for a minimum of 7 years (configurable)
  • Automated archiving of old data to secondary storage
  • Data integrity checks run daily to ensure no loss or corruption
  • Storage per sensor/year: ~10 MB (at 60-second logging)
Integrity Checks and Security

The system implements multiple layers of data integrity and security:

  • Database transaction logging enabled for recovery
  • Regular database backups and validation checks
  • Access control at the database level to prevent unauthorized modification
  • Electronic signatures required for all data deletion/modification (except automated data deletion)
  • Checksum validation for data transmission integrity

9.2 System Restore Process

Restore Prerequisites
  • Stop the Environmental Monitoring Windows Service
  • Ensure the target database server is operational and accessible
  • Have the latest verified backup file available and accessible
  • Restore process requires Administrator credentials for the database server
  • Notify system users of scheduled downtime for restore operations
  • Document the restore plan including rollback procedures
Database Restore Steps (SQL Server)
  1. Open SQL Server Management Studio (SSMS) with administrative privileges
  2. Right-click the database > Tasks > Restore > Database
  3. Select "From device" and locate the backup file
  4. Ensure "Overwrite the existing database" is selected
  5. Verify restore options and file locations
  6. Execute the restore operation and monitor progress
  7. Check the restore log for successful completion
  8. Verify database integrity after restore completion

9.3 Recovery Validation

Validation Steps Post-Restore
  1. Start the Environmental Monitoring Windows Service
  2. Access the Web Application and log in with administrator credentials
  3. Verify a critical sensor's latest reading and historical trend
  4. Check the Audit Trail for the last 24 hours of activity
  5. Confirm that the system time and date are correct and synchronized
  6. Generate a historical report to validate data integrity and completeness
  7. Test alarm functionality by simulating a sensor alarm condition
  8. Verify user access and permissions are correctly restored
  9. Test data export functionality to ensure reports can be generated
Recovery Documentation Requirements

Complete the following documentation for all restore operations:

  • Document the date and time of the restore operation
  • Record the backup file used for the restore (filename, size, timestamp)
  • Note any deviations or anomalies encountered during validation
  • Record system performance metrics post-restore
  • Document any data loss or inconsistencies identified
  • Obtain electronic signature from IT and QA departments for recovery sign-off
  • Update the system recovery log and incident documentation

  4. Apply the update package and run migration scripts (if any)
  5. Restart services and perform post-update validation
  6. Document the update in the change control system
  7. Notify users of the update and any changes to functionality
  8. Update system documentation and training materials as needed
Update Precautions
  • Never apply updates during active production monitoring
  • Always test updates in a non-production environment first
  • Ensure compatibility with existing hardware and configurations
  • Have a rollback plan in case the update causes issues
  • Document all update activities for audit trail and compliance
